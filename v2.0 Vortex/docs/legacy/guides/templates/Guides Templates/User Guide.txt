1. What this thing is (orientation)

The opening answers three questions fast:
What is this?
Who is it for?
What problem does it solve?

No marketing poetry. No buzzwords. A calm, grounding paragraph that lets the reader say, “Yes, I’m in the right place.”

2. What you need before you begin (prerequisites)

This is where you prevent rage later.

Devices, accounts, permissions, system requirements, skills assumed. If something must already exist, say so plainly. Hidden prerequisites are the number one cause of “this product is broken” emails.

3. First contact (installation or access)

This is the sacred moment.

Step-by-step, literal, boring in the best way:
How to install it, open it, or log in for the first time.
What the user should expect to see.
What success looks like at the end of this section.

If anything might go wrong here, warn them gently before it happens.

4. The mental model (how it works, simply)

Before buttons, explain the idea.

What are the main parts?
How do they relate?
What does the system do automatically vs what the user controls?

This is not a technical spec. It’s a story that lets the user build a map in their head so later steps make sense instead of feeling like magic rituals.

5. The first win (quick start / basic task)

Get the user to succeed fast.

One common, valuable task.
Minimal options.
Clear steps.
A visible result.

This section should create confidence. Once someone gets a win, they forgive imperfections everywhere else.

6. Core features (day-to-day use)

Now you expand.

Each major feature gets:
What it’s for (in human terms)
When you’d use it
How to use it
What to watch out for

This is where clarity beats completeness. Better to explain fewer things well than everything vaguely.

7. Settings, preferences, and customization

Only after the user understands normal use.

Explain:
What can be changed
Why someone might change it
What happens if they do

Always include how to reset to defaults. That sentence alone prevents support tickets.

8. Errors, problems, and recovery

This section builds trust.

Common mistakes.
Error messages explained in plain language.
What the user should do next.
When it’s safe to stop and when it’s not.

Never shame the user. If something failed, the guide failed first.

9. Safety, privacy, and responsibility

Especially important for anything involving data, money, security, or automation.

What the product does with user data.
What it does not do.
What the user is responsible for.
Any irreversible actions clearly flagged.

This is where calm honesty matters more than reassurance.

10. Getting help and going further

Close the loop.

Where to find help.
How to report issues.
Where updates or advanced docs live.
What to read next if they want to go deeper.

End with empowerment, not dependency.

The hidden rule most guides miss

A user guide is not written for you on your best day.
It’s written for someone tired, frustrated, and slightly suspicious at 11:47 pm.

If a sentence would confuse that person, rewrite it.

Do that consistently, and your guide stops being documentation and becomes infrastructure.
