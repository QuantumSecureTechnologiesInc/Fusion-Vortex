Documentation Writing GuidelinesThe Prime Directive: Documentation is not a chore; it is part of the product. We write "Code as Literature." Our documentation must be narrative-driven, human-centric, and clear.1. The Author's MindsetBefore filling out any template, adopt the following mindset:Narrative Arc: Every guide has a beginning (The Context), a middle (The Action), and an end (The Success). Do not just list facts; tell the user's story.Human-to-Human: Write as if you are a senior engineer sitting next to a junior colleague. Use "You" and "We." Be direct but kind.The "Why" over the "What": Don't just explain what a command does; explain why the user needs to run it.British English: Use standard British spelling (e.g., Colour, Organise, Optimisation, Centre, Behaviour).2. Filling Out the TemplatesA. The Developer Guide (DEVELOPER_GUIDE.md)Target Audience: Future Contributors and Colleagues.The Philosophy: This is the most important section. Do not skip it. Replace the placeholder with a passionate defence of why this project exists. Is it for speed? Safety? Simplicity? Define the project's soul here.The Toolchain: Be specific with versions. "Node.js" is insufficient; write "Node.js v20.10+ (LTS)".The PR/FAQ: If your project doesn't strictly use this method, explain your alternative. The goal is to enforce design thinking before coding.B. The Product Guide (docs/PRODUCT_GUIDE.md)Target Audience: Stakeholders, Product Managers, and Architects.Executive Summary: This is your elevator pitch. If you had 30 seconds to explain the product to a CTO, what would you say?Scenarios: These must be realistic. Avoid "Foo/Bar" examples. Use "The Enterprise Audit" or "The Startup MVP" to ground the features in reality.Benefit Statements: When listing capabilities, always pair them with a benefit.Bad: "It has a cache."Good: "Integrated caching reduces database load by 40%, lowering infrastructure costs."C. The User Guide (docs/USER_GUIDE.md)Target Audience: The End User (Developer or Operator).The Vision: Replace the abstract with a "Press Release" style hook. Promise a specific emotional result (relief, confidence, speed).The 5-Minute Success: This is critical. The commands you insert here must work copy-paste without modification.Do not leave [Insert Command] unless it is truly variable.If possible, provide a curl | bash script or a specific npm init command.Diátaxis Strictness:Tutorials: Step-by-step, linear, no choices.How-To: Problem/Solution format.Reference: Dry, factual tables.Explanation: Discursive, architectural context.D. The Info Sheet (docs/INFO_SHEET.md)Target Audience: Decision Makers (Marketing/Sales contexts).Brevity is King: Keep this to one printed page.The Big Three: Pick exactly three features. No more. If everything is a priority, nothing is.Metrics: Use real or estimated targets. If you don't have hard data, use design targets (e.g., "Designed for < 50ms latency").E. The Technical Sheet (docs/TECHNICAL_SHEET.md)Target Audience: SysAdmins and Security Auditors.Precision: Do not use "Recent Linux". Use "Linux Kernel 5.4+".Compliance: If you are not compliant with a standard (e.g., SOC2), list it as "Roadmap" or remove it. Do not lie.Dependencies: List everything required to run the binary, including libc versions if relevant for Linux.3. ### The "Plain English" Check
Before publishing, read your content aloud. If you stumble over a sentence, rewrite it.
* *Complex:* "Utilisation of the cache functionality is necessitated for optimisation."
* *Better:* "You must use the cache to improve performance."

## 4. Final Review Checklist

* [ ] **Completeness:** Are all `[Placeholders]` removed?
* [ ] **Tone:** Is it active voice? ("Run this command" vs "This command should be run")
* [ ] **Spelling:** Is British English used consistently?
* [ ] **Links:** Do the links to `CONTRIBUTING.md` etc. work relative to the file location?
* [ ] **Truth:** Did you actually run the "5-Minute Success" commands on a fresh machine?

> **Remember:** Good documentation is the difference between a tool people *use* and a tool people *love*.
