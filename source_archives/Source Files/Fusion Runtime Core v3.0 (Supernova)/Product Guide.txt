Product Guide: Fusion Runtime Core v3.0 "Supernova"1. Design PhilosophySupernova was born from the observation that generic async runtimes (like Tokio or async-std) are optimized for network I/O, not compute heterogeneity. When a GPU kernel finishes, standard runtimes require a slow context switch or a blocking wait. Supernova introduces a Unified Event Loop where hardware interrupts, network packets, and WASM trap exits are all handled by the same lock-free ring buffer.2. Core Features2.1 The Hyper-Ring ReactorInstead of epoll or kqueue, Supernova uses a userspace ring buffer similar to io_uring.Submission Queue (SQ): Pushes tasks (CPU futures, GPU kernel launches, WASM calls).Completion Queue (CQ): Hardware interrupts write directly here.Benefit: Zero syscall overhead for dispatching work to accelerators.2.2 Native WASM IntegrationWASM is not an afterthought; it is a Tier-2 citizen.Host Functions: WASM modules can directly invoke fusion_gpu_launch via high-performance trampoline functions.Shared Memory: The runtime maps GPU-pinned memory directly into the WASM linear memory space (read-only) for zero-copy inference.2.3 Distributed "Mesh" ModeNodes in a Supernova cluster form a mesh.Actor Model: Every task has a globally unique ID.Location Transparency: runtime.spawn_distributed(task) automatically places the task on the node with the lowest load or specific hardware (e.g., "Node with A100 GPU").State Migration: Running WASM tasks can be paused, serialized (stack + heap), and resumed on a different node.3. The Three-Tier ArchitectureTier 1: The Metal (Native)Language: Rust.Access: Full system access.Use Case: Core engines, drivers, heavy lifting.Tier 2: The Sandbox (WASM)Language: Any WASM-target (Rust, C++, AssemblyScript).Access: Capability-gated.Use Case: User scripts, business logic, multi-tenant functions.Tier 3: The Cloud (Distributed)Protocol: Custom FusionProto over UDP/QUIC.Access: Network-gated.Use Case: Scaling out to thousands of cores.4.
